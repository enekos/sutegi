# Migrations

sutegi's migration system turns a change to a `#[derive(Model)]` struct into a
versioned, reversible, backend-portable migration — the TypeORM *generate*
workflow without TypeORM's live-database nondeterminism.

There are two kinds of migration:

- **Declarative (ops) migrations** — a list of `SchemaOp`s (create table, add
  column, …) stored as a JSON file. Generated by diffing your models against the
  migration history, reversible for free, and portable across SQLite and
  Postgres.
- **Closure migrations** — hand-written `up`/`down` closures for data backfills
  and DDL the diff engine doesn't model. The escape hatch.

## The core idea: diff against history, not the live DB

`migrate:gen` computes the changes by diffing your models against the **shadow
schema** — the schema you get by folding every existing migration's ops in
memory. It never looks at your live database to decide what to generate.

That's the whole ballgame for determinism: the same repository state always
generates the same migration, no matter what anyone's local database happens to
contain. (TypeORM diffs against the live DB, which is why its generated
migrations depend on machine state and can silently emit `DROP COLUMN`.)

The live database is consulted for exactly one thing: **drift detection**.

## The workflow

```rust
#[derive(Model)]
struct Todo {
    #[model(primary)] id: i64,
    title: String,
}

fn migrations() -> Migrator {
    Migrator::new().load_dir(sutegi::migrate::MIGRATIONS_DIR).expect("load migrations")
}

fn main() -> std::io::Result<()> {
    let db = Db::open_or_memory("DATABASE_PATH");
    let models = sutegi::schemas![Todo];   // the desired state
    if sutegi::migrate::dispatch_full(&migrations(), &db, &models, sutegi::migrate::MIGRATIONS_DIR) {
        return Ok(()); // a migrate subcommand ran
    }
    migrations().run(&db).expect("migrate");
    // … serve …
}
```

Then, from the shell:

```
myapp migrate:gen create_todos   # writes migrations/<ts>_create_todos.json
myapp migrate                    # applies pending migrations
```

Add a field to the model:

```rust
#[derive(Model)]
struct Todo {
    #[model(primary)] id: i64,
    title: String,
    #[model(default = false)] done: bool,   // NEW
}
```

```
myapp migrate:gen add_done       # diffs Todo vs the shadow → one AddColumn
myapp migrate                    # applies it; existing rows get the default
```

This is the case the old `Model::migrate` got wrong: `CREATE TABLE IF NOT EXISTS`
silently ignored the new column, and inserts began failing. Now the change is a
reviewed, reversible migration.

## CLI verbs (`dispatch_full`)

| Command | What it does |
|---|---|
| `migrate` | apply all pending migrations |
| `migrate:rollback [n]` | roll back the last `n` batches (default 1) |
| `migrate:status` | print the ledger (✓ applied, ? orphan) |
| `migrate:gen <name>` | diff models ↔ shadow, write a migration file |
| `migrate:plan` | show what `:gen` would write, without writing |
| `migrate:drift` | three-way report: DB vs migrations vs models |
| `migrate:fresh` | roll everything back and re-run (dev only) |

`dispatch` (without `_full`) is the smaller runner: just `migrate` /
`:rollback` / `:status`, for apps that hand-write migrations.

## Column attributes

The generated schema comes from `#[derive(Model)]`:

```rust
#[derive(Model)]
struct Post {
    #[model(primary)] id: i64,
    #[model(unique)] slug: String,
    #[model(index)] author_id: i64,
    #[model(default = 0)] views: i64,
    #[model(belongs_to(User, foreign_key = "author_id", on_delete = "cascade"))]
    author: Option<User>,
}
```

`unique`, `index`, `default = <lit>`, and `belongs_to(..., on_delete = ...)` all
flow into the schema IR, so the diff sees uniques, secondary indexes, defaults,
and foreign keys.

## Safety and destructive changes

Every op is classified:

- **Safe** — create table, add a nullable/defaulted column, create an index, add
  a foreign key, widen a column.
- **NeedsData** — add a `NOT NULL` column with no default, or tighten a column to
  `NOT NULL`. Valid only against an empty table; give it a `.default(...)` or add
  a backfill.
- **Destructive** — drop a table/column, a lossy type change.

`migrate:gen` writes destructive ops but flags them loudly in the output; the
committed, reviewed file is your approval. **Renames are never guessed**: a
dropped column plus an added column of the same type generate a `DropColumn` +
`AddColumn` pair with a "possible rename?" warning — edit the file to a
`RenameColumn` if that's what you meant (which preserves the data).

## Dev-mode sync

For fast local iteration, `Model::migrate` (and `sutegi::orm::migrate::sync`)
apply **additive, non-destructive** changes directly — create missing tables, add
columns/indexes/foreign keys — without writing a migration file. They never drop
anything, and they refuse (pointing you at `migrate:gen`) any change that needs a
real migration. Use this locally; use migration files in production.

## Drift detection

`migrate:drift` compares three things:

- **DB vs migrations** — the live schema diverges from what the migrations say it
  should be (a hand-edit, or migrations not fully applied).
- **Models vs migrations** — you changed a model but haven't run `migrate:gen`.

A read-only JSON view is available for agents/dashboards via
`sutegi::migrate::report_json(&migrator, &db, &models)`, which you can mount:

```rust
let report = sutegi::migrate::report_json(&migrations(), &db, &models);
let app = app.get("/__migrations", "Migration status + drift.",
    move |_| sutegi::web::json(200, &report));
```

## Integrity: checksums

The `_sutegi_migrations` history table stores a checksum of each applied ops
migration. If a migration file is edited after being applied, `migrate` fails
with a checksum mismatch rather than silently diverging. After a deliberate edit,
`Migrator::repair(&db)` re-stamps the stored checksums.

## Cross-pod safety

On Postgres, `run`/`rollback` take a session advisory lock, so many pods booting
at once serialize their migrations instead of racing. SQLite is single-node, so
this is a no-op there.

## Honest caveats

- `down` restores schema **shape, not data** — a dropped column comes back empty.
- The schema IR covers tables, columns (type/nullability/unique/default/primary),
  secondary indexes, and single-column foreign keys. Views, triggers, sequences,
  CHECK constraints, and multi-column FKs are out of scope; use a closure
  migration for those.
- A `vector(dim)` column reflects back dimensionless from Postgres introspection
  (the dimension lives in a catalog the reflection doesn't read), so a dimension
  change won't be detected by drift. Declare it in a migration explicitly.
- On SQLite, changing a column's type/nullability or altering a foreign key is
  done with a full table rebuild (create-new / copy / drop / rename). This
  preserves rows but is heavier than an in-place `ALTER` — Postgres does those in
  place.
